# Runbook: Organization Annual Entitlement — Admin Grant, Extend, Revoke

**Scope:** Operator SSoT for platform-admin management of organization annual entitlement in Cardigo.
**Status:** Active — last reviewed 2026-04-26.
**Tier:** Tier 2 (Architecture / Ops Contracts) per `copilot-instructions.md`.

---

## 1) Purpose

Organization annual entitlement is the mechanism by which a platform admin grants premium access to all org-owned cards under a specific organization, for an agreed annual period.

Key properties:

- **Platform-admin-managed only.** This is not self-service.
- **B2B / offline-payment model.** The operator collects payment externally (invoice, bank transfer, contract), then grants access manually via the admin UI.
- **Annual.** Entitlement has an explicit start and expiry date.
- **Organization-scoped.** All org-owned cards under an active entitlement receive premium automatically through the computed resolver — no per-card action needed.

---

## 2) Product Decision

Org cards are not sold through the ordinary personal checkout flow (`/payment/checkout`).

The business model for organizations is:

1. Platform admin creates the org (slug, name, seat limit).
2. Platform admin agrees terms/payment externally with the client (invoice, transfer, CRM).
3. Platform admin grants annual entitlement via the admin panel after confirmed payment.
4. All org-owned cards immediately receive premium entitlements through resolver.
5. When entitlement approaches expiry, the operator renews it manually via Extend.

This separation is intentional:

- Avoids forcing B2B clients through a consumer Tranzila hosted checkout.
- No STO/recurring billing for org contracts (handled manually).
- No automatic receipts/invoices generated from the entitlement grant.
- Organization billing is fully operator-mediated.

## 4) When to Grant

**Only after confirmed external payment or signed agreement.**

Do not grant speculatively. The entitlement is not a trial or a preview — it represents a paid/contracted annual access period.

Operator checklist before granting:

- [ ] External payment confirmed (bank transfer, invoice paid, contract signed).
- [ ] Org is active (`isActive: true`).
- [ ] Correct `orgId` identified in admin UI.
- [ ] `expiresAt` set to the correct contract end date.
- [ ] `reason` includes the reference context (e.g., invoice number, contract note).
- [ ] `paymentReference` optionally set to an external reference ID (max 120 chars).
- [ ] `adminNote` optionally set for internal context (max 500 chars).
- [ ] Confirmation checkbox checked (acknowledgement gate).

**Important:** `paymentReference` and `adminNote` are **operator notes only**. They are stored in the org entitlement subdocument for audit purposes. They are not payment provider data, are not sent to any provider, and do not generate any invoice or receipt.

## 6) Data Model / SSoT

### Organization.orgEntitlement

`Organization.orgEntitlement` is the SSoT for org annual entitlement.

Conceptual fields (server-side):

| Field              | Type           | Description                                                                    |
| ------------------ | -------------- | ------------------------------------------------------------------------------ |
| `status`           | String enum    | `"none"` \| `"active"` \| `"revoked"`                                          |
| `plan`             | String         | `"org"`                                                                        |
| `source`           | String         | `"admin-manual"` (stored value; resolver emits `"organization"` as DTO source) |
| `startsAt`         | Date \| null   | Entitlement start date (optional in UI V1)                                     |
| `expiresAt`        | Date \| null   | Entitlement expiry date                                                        |
| `grantedAt`        | Date           | Timestamp of last grant operation                                              |
| `lastModifiedAt`   | Date           | Timestamp of last modification                                                 |
| `paymentReference` | String \| null | Operator payment reference (max 120 chars)                                     |
| `adminNote`        | String \| null | Operator internal note (max 500 chars)                                         |

### Resolved DTO

When the billing resolver evaluates an org-owned card under an active entitlement, it returns:

```json
{
    "source": "organization",
    "isPaid": true,
    "plan": "org"
}
```

**Important terminology:**

- `effectiveBilling.source = "organization"` is the public/resolved DTO value.
- `orgEntitlement` is the internal Organization model field name.
- Do not confuse these. Never expect DTO `source = "orgEntitlement"` or `source = "org-entitlement"`.

Internal audit fields (`paymentReference`, `adminNote`, `grantedAt`, etc.) are not included in public card DTOs.

## 7) State Machine

```
                    ┌──────────────────────┐
                    │         none         │
                    │  (no record or       │
                    │   status=none)       │
                    └──────────┬───────────┘
                               │ admin Grant
                               ▼
                    ┌──────────────────────┐     expiresAt > now
                    │        active        │ ────────────────────► effectively active
                    │  status=active       │
                    │                      │     expiresAt <= now
                    │                      │ ────────────────────► expired (computed)
                    └────────┬─────────────┘
                             │ admin Revoke
                             ▼
                    ┌──────────────────────┐
                    │       revoked        │
                    │  status=revoked      │
                    └──────────────────────┘
```

**State definitions:**

| State                | How determined                              | Effect                                                     |
| -------------------- | ------------------------------------------- | ---------------------------------------------------------- |
| `none`               | No entitlement record, or `status = "none"` | All org cards resolve to free                              |
| `active`             | `status = "active"` AND `expiresAt > now`   | All org cards receive premium                              |
| `expired` (computed) | `status = "active"` AND `expiresAt <= now`  | Computed client-side for display; cards fall back to free  |
| `revoked`            | `status = "revoked"`                        | All org cards resolve to free; historical fields preserved |

**Notes:**

- `expired` is a computed display state, not necessarily a distinct stored status value. The resolver checks `status === "active"` AND `expiresAt > now` to determine effective premium access.
- `revoked` preserves historical `plan`, `startsAt`, `expiresAt`, and audit fields — they remain in the record for audit/history.
- After revocation, a new Grant operation starts a fresh entitlement cycle.
- An expired-active record is shown as "expired" in the UI and offers the Grant action.

**Admin UI visibility rules:**

| Computed state | "Grant" button shown | "Extend" button shown | "Revoke" button shown |
| -------------- | -------------------- | --------------------- | --------------------- |
| `none`         | ✓                    | ✗                     | ✗                     |
| `active`       | ✗                    | ✓                     | ✓                     |
| `expired`      | ✓                    | ✗                     | ✗                     |
| `revoked`      | ✓                    | ✗                     | ✗                     |

## 8) Admin UI Operator Flow

### Accessing the panel

1. Navigate to `/admin`.
2. Select **Organizations** from the left navigation.
3. Click on the target organization to open the detail panel.
4. Scroll to the **"הרשאה שנתית לארגון"** (Organization Annual Entitlement) section.

### Status badge

The badge at the top of the entitlement section shows computed status:

| Badge       | Color  | Meaning                                   |
| ----------- | ------ | ----------------------------------------- |
| `ללא הרשאה` | Grey   | No entitlement                            |
| `פעיל`      | Green  | Active (expiresAt > now)                  |
| `פג תוקף`   | Orange | Expired (status=active, expiresAt <= now) |
| `בוטל`      | Red    | Revoked                                   |

The details grid below the badge shows: status, plan, startsAt, expiresAt, source, grantedAt, lastModifiedAt.

### Grant flow

1. Click **"הענק גישה שנתית"** (Grant Annual Access).
2. A form panel opens.
3. Fill fields:
    - **Expiry date** (required): the annual contract end date.
    - **Reason** (required, 5–500 chars): context for the grant (invoice number, agreement note).
    - **Payment reference** (optional, max 120 chars): external reference ID.
    - **Admin note** (optional, max 500 chars): internal context.
4. Check the confirmation checkbox: **"אני מאשר הענקת גישה שנתית לאחר תשלום/הסכם מחוץ למערכת"**.
5. Click **"הענק גישה"** (Grant).
6. On success: status badge updates; details grid reflects new values; Grant button replaced by Extend + Revoke.

### Extend flow

1. Click **"הארך גישה"** (Extend Access).
2. A form panel opens. Current expiry is displayed.
3. Fill fields:
    - **New expiry date** (required, must be after current `expiresAt`).
    - **Reason** (required, 5–500 chars).
    - **Payment reference** (optional).
    - **Admin note** (optional).
4. Click **"הארך גישה"** (Extend).
5. On success: `expiresAt` and `lastModifiedAt` update.

### Revoke flow

1. Click **"בטל גישה"** (Revoke Access).
2. A form panel opens with a warning that org cards will lose premium access.
3. Fill fields:
    - **Reason** (required, 5–500 chars).
4. Check the confirmation checkbox: **"אני מאשר ביטול הגישה השנתית לארגון זה"**.
5. Click **"בטל גישה"** (Revoke).
6. On success: status badge shows "בוטל" (Revoked); Grant button reappears.

### Offline payment disclaimer

The panel includes a static notice:

> "לא מתבצעת כאן סליקה, יצירת חשבונית או חיוב אוטומטי."
> (No payment processing, invoice creation, or automatic billing occurs here.)

## 9) Backend API Contract

All endpoints require platform-admin auth (`user.role === "admin"`).

### GET /api/admin/orgs/:id

Returns the organization detail DTO including `entitlement` subdocument.

```json
{
    "entitlement": {
        "status": "active",
        "plan": "org",
        "source": "admin-manual",
        "startsAt": null,
        "expiresAt": "2027-04-26T00:00:00.000Z",
        "grantedAt": "2026-04-26T...",
        "lastModifiedAt": "2026-04-26T..."
    }
}
```

### POST /api/admin/orgs/:id/entitlement/grant

**Required body fields:**

| Field                   | Type            | Rules                           |
| ----------------------- | --------------- | ------------------------------- |
| `expiresAt`             | ISO 8601 string | Required; must be a future date |
| `reason`                | String          | Required; 5–500 chars           |
| `confirmOrgAnnualGrant` | Boolean         | Must be `true`                  |

**Optional body fields:**

| Field              | Type            | Rules                      |
| ------------------ | --------------- | -------------------------- |
| `startsAt`         | ISO 8601 string | Optional; omitted by UI V1 |
| `paymentReference` | String          | Optional; max 120 chars    |
| `adminNote`        | String          | Optional; max 500 chars    |

**Success:** `200 { ok: true }` (or `{ ok: true, auditWriteFailed: true }` — see §11)

### POST /api/admin/orgs/:id/entitlement/extend

**Required body fields:**

| Field          | Type            | Rules                                       |
| -------------- | --------------- | ------------------------------------------- |
| `newExpiresAt` | ISO 8601 string | Required; must be after current `expiresAt` |
| `reason`       | String          | Required; 5–500 chars                       |

**Optional body fields:**

| Field              | Type   | Rules                   |
| ------------------ | ------ | ----------------------- |
| `paymentReference` | String | Optional; max 120 chars |
| `adminNote`        | String | Optional; max 500 chars |

**Success:** `200 { ok: true }`

### POST /api/admin/orgs/:id/entitlement/revoke

**Required body fields:**

| Field    | Type   | Rules                 |
| -------- | ------ | --------------------- |
| `reason` | String | Required; 5–500 chars |

**Success:** `200 { ok: true }`

## 10) Error Codes and Operator Meaning

| HTTP  | Code                              | Meaning                                                                      | Operator action                  |
| ----- | --------------------------------- | ---------------------------------------------------------------------------- | -------------------------------- |
| `409` | `ENTITLEMENT_ALREADY_ACTIVE`      | Grant attempted when entitlement is already active.                          | Use Extend instead.              |
| `409` | `NOT_ACTIVE`                      | Extend attempted but no active entitlement exists.                           | Grant first.                     |
| `409` | `NO_ENTITLEMENT`                  | Revoke attempted but nothing to revoke.                                      | No action needed.                |
| `409` | `INACTIVE_ORG`                    | Target org is not active (`isActive: false`).                                | Activate org first, then grant.  |
| `400` | `CONFIRM_REQUIRED`                | `confirmOrgAnnualGrant` was `false` or missing.                              | Check the confirmation checkbox. |
| `400` | `INVALID_REASON`                  | Reason was less than 5 or more than 500 chars.                               | Fix the reason field.            |
| `400` | `INVALID_EXPIRES_AT`              | `expiresAt` is missing, not a valid ISO date, or not in the future.          | Provide a valid future date.     |
| `400` | `INVALID_DATE_RANGE`              | `newExpiresAt` is not after current `expiresAt` on extend.                   | Provide a later date.            |
| `400` | `INVALID_PAYMENT_REFERENCE`       | `paymentReference` exceeds 120 chars.                                        | Shorten the reference.           |
| `400` | `INVALID_ADMIN_NOTE`              | `adminNote` exceeds 500 chars.                                               | Shorten the note.                |
| `409` | `ORG_CARD_BILLING_MANAGED_BY_ORG` | Returned when attempting legacy billing/trial operations on org-owned cards. | See §15 (Admin Hardening).       |

## 11) AdminAudit Log

Every successful entitlement write generates an `AdminAudit` record.

| Operation | Action value             |
| --------- | ------------------------ |
| Grant     | `ORG_ENTITLEMENT_GRANT`  |
| Revoke    | `ORG_ENTITLEMENT_REVOKE` |
| Extend    | `ORG_ENTITLEMENT_EXTEND` |

Each record includes: admin user ID, action, target org ID, reason, and before/after entitlement state.

### `auditWriteFailed: true`

If the entitlement write succeeds but the AdminAudit log write fails, the response is:

```json
{
    "ok": true,
    "auditWriteFailed": true,
    "warning": "..."
}
```

**This means the entitlement operation succeeded. Only the audit log write failed.**

Operator action:

- Do **not** retry the entitlement operation (that would double-grant / double-extend).
- Inspect backend logs to reconstruct the operation for manual audit trail.
- Record the event manually if needed.
- The org entitlement state in the DB is authoritative.

## 12) Resolver Behavior

When the billing resolver evaluates a card:

1. If `card.orgId` is set (org-owned card) AND the organization has `orgEntitlement.status === "active"` AND `orgEntitlement.expiresAt > now`:
    - `effectiveBilling.source = "organization"`
    - `effectiveBilling.plan = "org"`
    - `effectiveTier = "premium"`
    - `PLANS.org` feature gates apply (publish, seo, analytics, slugChange, leadForm, video, reviews, templates:"all", gallery, galleryLimit:50)
2. Otherwise (expired, revoked, or no entitlement): org-owned cards fall back through the normal resolver chain (personal billing → free).

**What is NOT affected:**

- Personal cards belonging to org members — these continue to use personal billing/trial/free logic.
- `User.subscription` fields — never mutated by org entitlement operations.
- Cards not owned by the org — org entitlement is strictly scoped to `card.orgId`.

---

## 13) AI Behavior

Org entitlement grants access to the premium AI profile for org card AI operations.

- Org-owned cards under active entitlement receive the `monthly`/`yearly` AI profile (30 generations/month shared budget).
- **Current V1:** Quota is user-scoped, not org-scoped. There is no org-wide AI quota pool.
- Per-user monthly quota is enforced via `AiUsageMonthly` by `userId`.
- Future: org-wide AI quota pool is a deferred non-goal of V1.

---

## 14) Job Safety

The following scheduled/background jobs skip org-owned cards by design:

| Job                       | Behavior                                                                           |
| ------------------------- | ---------------------------------------------------------------------------------- |
| `retentionPurgeJob`       | Skips cards where `card.orgId` is set. Org-owned cards are never retention-purged. |
| `trialLifecycleReconcile` | Skips org-owned cards. Org-owned cards do not enter the personal trial lifecycle.  |

**Why:** Org cards are governed by `Organization.orgEntitlement`, not personal trial/retention lifecycle. The jobs are scoped to personal-card flows only.

## 15) Admin Hardening

The following legacy admin billing/trial operations are **blocked** for org-owned cards:

| Operation                              | Endpoint / Function                                           | Block code                            |
| -------------------------------------- | ------------------------------------------------------------- | ------------------------------------- |
| Extend trial                           | `POST /api/admin/cards/:id/trial/extend`                      | `409 ORG_CARD_BILLING_MANAGED_BY_ORG` |
| Set card billing (adminSetCardBilling) | Admin card billing override endpoint                          | `409 ORG_CARD_BILLING_MANAGED_BY_ORG` |
| Sync card billing from user            | `adminSyncCardBillingFromUser`                                | `409 ORG_CARD_BILLING_MANAGED_BY_ORG` |
| Revoke card billing                    | `adminRevokeCardBilling` (via adminSetCardBilling delegation) | `409 ORG_CARD_BILLING_MANAGED_BY_ORG` |

**File truth:** `backend/src/controllers/admin.controller.js` — `isNonPersonalOrgCard()` helper and guards at `extendTrial`, `adminSetCardBilling`, `adminSyncCardBillingFromUser` (Phase 2H).

**adminOverride / overridePlan** is a deferred escape hatch and was intentionally **not blocked** in Phase 2H. If a future use case requires bypassing org entitlement for a single card, `adminOverride` remains available as a platform-admin escape hatch under explicit review. It is not the org annual entitlement grant mechanism.

## 18) Troubleshooting

### Entitlement shows "expired" immediately after grant

- **Cause:** `expiresAt` was set in the past.
- **Fix:** Revoke and re-grant with a future `expiresAt`.

### Grant returns `ENTITLEMENT_ALREADY_ACTIVE`

- **Cause:** The org already has an active entitlement (`status=active`, `expiresAt > now`).
- **Fix:** Use Extend instead of Grant.

### Extend returns `NOT_ACTIVE`

- **Cause:** No active entitlement exists (none, expired computed, or revoked).
- **Fix:** Use Grant first to create a new entitlement.

### Revoke returns `NO_ENTITLEMENT`

- **Cause:** No entitlement record exists to revoke.
- **Fix:** No action needed (already no entitlement).

### `auditWriteFailed: true` in response

- **Cause:** Entitlement write succeeded, but the AdminAudit log write failed (e.g., transient DB issue).
- **Fix:** Do NOT retry. Inspect backend logs. Record event manually for audit trail.
- The org entitlement in MongoDB is authoritative — the operation completed successfully.

### Org card still free after successful grant

- **Cause:** Possibly page cached or stale DTO in browser.
- **Fix:** Hard-refresh the admin card detail page or call `GET /api/admin/cards/:id` directly to see fresh DTO.
- If DTO still shows free: verify org `orgEntitlement.status === "active"` and `expiresAt > now` in MongoDB.
- Confirm `card.orgId` is correctly set on the card.

### Admin tries payment checkout for org card

- **Symptom:** Org client has org card but tries to upgrade via `/payment/checkout`.
- **Policy:** Org cards are not sold through personal checkout. Redirect the operator to grant org annual entitlement through the admin panel.

### Admin tries legacy trial/billing operation on org card

- **Symptom:** `POST /api/admin/cards/:id/trial/extend` returns `409 ORG_CARD_BILLING_MANAGED_BY_ORG`.
- **Fix:** This is expected behavior. Org cards are managed via org annual entitlement. Do not attempt trial extension for org cards.

## 19) Explicit Non-Goals (V1)

These are **not** implemented and are not planned in the current entitlement contour:

- Self-service org checkout (no org-admin-initiated checkout flow)
- Automatic recurring org billing (no STO/recurring for org plans)
- Payment provider integration for org entitlement (no Tranzila, no STO, no YeshInvoice)
- Automatic receipt/invoice generation from grant/extend operations
- Org-wide AI quota pool (per-user quota only in V1)
- Public/user-facing entitlement status display (admin panel only)
- Org-admin self-service entitlement management (platform admin only)
- Org entitlement expiry notification / webhook
