Auth: Forgot/Reset Password — Runbook (Enterprise)

Purpose
Password reset via email token. Token is delivered only by email. API must not leak whether an account exists.

Endpoints

POST /api/auth/forgot

Input: { "email": "<user email>" }

Output: 204 No Content always (anti-enumeration), regardless of whether the email exists or is valid.

Side effects (live as of 2026-03-29):

If user exists, backend performs two durable DB writes before responding:

1. Upserts ActivePasswordReset (status: pending-delivery, no token yet) — one-active-per-user via unique userId index (findOneAndReplace).
2. Creates MailJob (userId only — no email, no token snapshot).
   resetMailWorker then picks up the MailJob on its next tick, generates rawToken only in memory at delivery time, sets tokenHash on ActivePasswordReset (status: active), builds the reset link, and sends via Mailjet.

Single-active token semantics: the findOneAndReplace upsert atomically supersedes any prior ActivePasswordReset for the user — no separate invalidation step required.

Rate limit baseline applies; exceeding returns 204 (not 429) to preserve anti-enumeration posture.

POST /api/auth/reset

Input: { "token": "<raw token>", "password": "<new password>" }

Output:

204 No Content on success (password updated, token one-time used).

400 with neutral message on any failure (invalid/expired/used token, missing user, etc.). No reason leakage.

Token security model

Raw token is never stored in DB — not even transiently. ActivePasswordReset records have no tokenHash until resetMailWorker sets it immediately before calling Mailjet.

rawToken is generated by resetMailWorker (crypto.randomBytes(32)) in memory only within the delivery call frame; JS GC reclaims it after the frame returns.

DB stores tokenHash = sha256(rawToken) on ActivePasswordReset (status: active) once the worker reaches the delivery step.

MailJob stores userId only — no email snapshot, no token, no link.

One-time use enforced by atomic consume on ActivePasswordReset (findOneAndUpdate with status:'active' + usedAt:null guard).

Legacy transition: /auth/reset also falls back to the PasswordReset collection for tokens issued before the runtime cutover. All legacy tokens expire within 30 min post-cutover. Remove this fallback in the next release cycle.

Tokens are time-limited (30 min TTL, RESET_TOKEN_TTL_MS).

resetMailWorker

The worker runs in-process, started by server.js alongside trialCleanupJob:

- Initial delay: 30 seconds after boot.
- Recurring interval: 60 seconds (configurable via RESET_MAIL_WORKER_INTERVAL_MS env var).
- Claim semantics: findOneAndUpdate({status:'pending'}) — FIFO by createdAt, atomic.
- User-missing path: MailJob → abandoned, ActivePasswordReset → used. No email sent.
- APR-not-claimable path: MailJob → abandoned, rawToken discarded.
- Success path: MailJob → sent.
- Not configured (no Mailjet keys): MailJob → failed.
- Ambiguous/exception path: MailJob left in processing (see Stuck processing jobs below).
- Sentry monitor slug: reset-mail-worker (active only if Sentry is configured).

Stuck processing jobs

First-version policy: the worker does NOT auto-retry with a regenerated token on ambiguous Mailjet outcome. A MailJob left in processing means:

- The link may or may not have been delivered.
- Overwriting tokenHash would invalidate a potentially working link without certainty.

Operator action for a stuck job:

1. Ask the user to submit a new /forgot request. The cooldown check detects no live MailJob (pending|processing) and allows a retry, creating a fresh APR and MailJob.
2. If the job must be manually cleared: set MailJob.status = 'abandoned' in the DB (and optionally ActivePasswordReset.status = 'used') to remove it from the queue.
3. Do not manually set rawToken or tokenHash — the worker is the sole token generator.

Mailjet semantics (best-effort)

Email sending is backend-only.

Mailjet failures do not break /forgot response (still 204).

No logging of raw tokens or full reset links.

Rate limiting baseline

/auth/forgot and /auth/reset are rate-limited (in-memory, IP-based).

Note: in-memory limiter is not shared across instances (enterprise future improvement: Redis/shared limiter).

/auth/forgot abuse hardening (live)

In addition to IP rate limiting, a backend-authoritative per-user DB-backed cooldown is enforced:

- Cooldown window: 180 seconds from creation of the last active unexpired reset token for the same userId.
- If a recent active token is found, the send is suppressed silently; 204 is returned (anti-enumeration preserved).
- Catch block is fail-closed: any DB error during cooldown check also returns 204 (error cannot bypass the abuse rail).
- IP rate-limit response is 204 (not 429): callers cannot distinguish IP limit from user-not-found or cooldown.
- Frontend client enforces a matching 180-second countdown UX with a spam/junk folder hint.

JWT session invalidation consequence

A successful /auth/reset call sets passwordChangedAt on the user record, which immediately invalidates all existing JWT sessions for that user. Any token with iat < passwordChangedAt.getTime()/1000 will be rejected by requireAuth (401) and silently dropped by optionalAuth on the next authenticated request.

Index governance

Mongoose autoIndex/autoCreate are disabled at runtime. All indexes must be applied manually via canonical migration scripts.

passwordresets collection (legacy — kept for dual-read transition window)

Canonical script: migrate:user-auth-indexes (backend/scripts/migrate-user-auth-indexes.mjs)

This script governs indexes for all auth token collections including passwordresets. It creates:

- tokenHash_1 (unique) — enforces at DB level that no two reset documents share a hash
- userId_1 — supports per-user cooldown and invalidation queries
- expiresAt_1 — supports expire-filter queries
- usedAt_1 — supports active-token queries

Operational procedure:

1. Dry-run first (default, safe, read-only):
   node scripts/migrate-user-auth-indexes.mjs
   or: npm.cmd run migrate:user-auth-indexes

2. Review output — confirm no BLOCKED messages.

3. Apply (mutates DB — run in a maintenance window):
   npm.cmd run migrate:user-auth-indexes -- --apply

4. Confirm post-check passes: script prints POST-CHECK: all critical indexes verified.

App-level expiry remains authoritative: even without DB indexes, the application always
filters on expiresAt: { $gt: now } and usedAt: null before accepting any token. Missing
indexes are a performance and uniqueness-guarantee gap, not a correctness or security gap.

TTL index decision (deferred):

A MongoDB TTL index on expiresAt (expireAfterSeconds: 0) would allow MongoDB to
automatically delete expired PasswordReset documents. This is NOT currently applied.

Current state: expired documents accumulate in the collection but are harmless — the
application-level expiry guard rejects them on every lookup.

To apply TTL in a future maintenance window (separate from the canonical index apply):
node scripts/migrate-passwordreset-indexes.mjs --apply --ttl
Note: if expiresAt_1 was already created as a plain index, it must be dropped before
recreating as a TTL index. Confirm with dry-run first.

activepasswordresets + mailjobs collections (applied 2026-03-29)

Canonical script: migrate:active-password-reset-indexes (backend/scripts/migrate-active-password-reset-indexes.mjs)

Status: APPLIED. Both collections and all indexes exist in the live DB.

Applied indexes — activepasswordresets:

- userId_1_unique (unique) — one-active-per-user structural guarantee
- tokenHash_1_partial_unique (unique, partial: tokenHash.$type=string) — excludes absent tokenHash
- expiresAt_1 — worker filter + cleanup
- status_1 — worker polling
- usedAt_1 — cleanup / audit

Applied indexes — mailjobs:

- userId_1 — per-user lookup
- status_1_expiresAt_1 (compound) — primary worker poll: {status:'pending', expiresAt:{$gt:now}}
- expiresAt_1 — cleanup sweep

Operational procedure (for drift check or future re-run):

Dry-run:
npm.cmd run migrate:active-password-reset-indexes -- --dry-run --verbose

Apply:
npm.cmd run migrate:active-password-reset-indexes -- --apply --verbose

Manual smoke (PowerShell + curl.exe)

Forgot: always 204:

send {email} and expect 204

Reset (token is generated by resetMailWorker, not inline by /forgot):

Wait for the worker tick (up to 60 s, or 30 s on first boot) before attempting /reset.

first use of token → 204

second use of same token → 400

Dual-read transition: tokens from the old PasswordReset collection (issued before runtime cutover) are also accepted for up to 30 minutes post-cutover.

Single-active:

run forgot twice → only the most recent APR + MailJob is valid; prior APR is atomically replaced by findOneAndReplace upsert.
